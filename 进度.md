## 当前有效基线（2026-07-14）

- 当前公开 API 版本来自 `nl_compat.lua`：`1.0.1`。
- 当前构建实际嵌入的是 `panorama_compat.lua` 纯 LuaJIT/FFI 后端；源码树中没有正在参与构建的 `src/plugin/panorama/panorama_bridge.cpp/.hpp`。
- 下方 0.9.x/1.0.x 条目保留为历史实验记录，不代表对应方案仍存在于当前构建；本节和顶部模块表优先级最高。
- 已验证的原生 GameEvent 只有 `player_hurt`、`player_chat`；其他事件虽然出现在 `events` 表中，但当前没有完整字段派发。
- `db` 当前只在单个 Lua 状态内保存 table，不是磁盘数据库。
- Panorama 当前只能视为实验接口。V8 Fatal Assertion 属于进程级终止，Lua `pcall/xpcall` 无法恢复。

### Panorama 当前使用原则

1. 优先使用 `panorama.run_script(js, panel)` 安装一次性的、完全驻留在 JS 内的逻辑。
2. 不要在 `events.render`、`events.tick` 或 `utils.execute_after` 中持续调用 `loadstring/open` 返回的代理。
3. 不要跨帧保存 JS Array/Object/Function/PersistentProxy。
4. 不要用 Panorama JS 回调直接调用 Lua，再从该 Lua 回调重入 Panorama。
5. `panorama_events` 不是正式 API；`panorama_test.lua` 的稳定实验版采用 JS-only 事件处理，只在安装时进入一次 Panorama。

### 0.9.19 Panorama V8 所属线程执行与退出崩溃修复

- 根因确认：DXGI Present 是渲染线程，不是 Panorama V8 isolate 所属线程；在 Present 调用 V8 会污染 isolate enter 状态，并在退出时触发 `Isolate::Dispose - Disposing the isolate that is entered by a thread`。
- 对已自动定位的 Panorama execute helper 安装 MinHook；Lua 请求只入队，不再从 Present/任意 Lua 调用线程直接进入 V8。
- 当游戏自身在正确的 Panorama 线程调用 execute helper 时，detour 排空队列，并通过原始 trampoline 执行脚本。
- 删除所有跨线程直接执行回退：线程钩子未就绪或超时只返回失败，绝不再冒险进入 isolate。
- 诊断新增 `v8_thread_hook_ready`。

### 0.9.18 Panorama 私有 ScriptEngine 修复

- 诊断新增 `script_engine` 地址。

### 0.9.15 Panorama CompileRunScript 槽位自动定位（已废弃）

- 实测 0.9.14 主线程编组后仍 `ok=true` 但 JS 不执行（SayParty/echo 均无效）→ 确认 `vtable[81]` 在当前 build 已不是 CompileRunScript（就像 GetID 从 slot 11 移到 64，槽位整体移位）。
- 改为**自动定位**：luv8 特征码 `E8 ?? ?? ?? ?? 48 8B D8 48 83 38 00 75 15` 位于 CompileRunScript 函数体内部（compile 调用 + 返回句柄判空）。取该特征码地址 P，在 UIEngine vtable 里找"函数入口 ≤ P 且最接近（P-A<0x8000）"的槽 → 即真正的 CompileRunScript 槽位，自适应任意 build。

### 0.9.14 Panorama JS 执行真正修复（主线程编组）

- 关键更正：对照 luv8 原始实现，`UIEngine vtable[81]` **就是** `nativeCompileRunScript`（编译并执行），调用形式 `(engine, panel, source, layout, 8, 10, false)`——和最初代码完全一致；`E8 ??..` 特征函数只是 `compile`（仅编译），0.9.13 用它反而是错的，已撤销。
- 真正根因是**线程亲和**：Panorama 的 V8 只能在游戏主线程执行；旧代码从插件线程调用 vtable[81]，V8 拒绝执行 → 返回非空但 JS 不跑（`ok=true` 无副作用）；0.9.13 从错误线程调用 compile 更导致 AV，回退后 isolate 处于异常态 → `ok=false`。
- 修复：新增主线程任务队列。`panorama.loadstring` / `run_script` 把"resolve+find+CompileRunScript"打包，通过 DXGI Present Hook 线程（游戏主线程）执行：
  - `LuaEngine::pump_main_thread()` 在 `HookPresent`/`HookPresent1` 开头记录主线程 id 并抽干任务队列。
  - `run_on_main_thread(fn)`：已在主线程则直接内联（UI 回调场景）；否则入队等待（默认 2s 超时防止卡死）。
  - 渲染事件由 `fire_event("render")` 改为非阻塞 `try_fire_event`，避免插件线程持锁时阻塞主线程 pump（防死锁）。
- vtable[81] 保留原 `(engine,panel,source,layout,8,10,false)` 调用；`debug_status` 仍打印 `run_script_found`（仅诊断，不用于执行）。
- 需要**重新注入新 DLL**（`build\Release\CS2LuaPlugin.dll` 19:23，1581056 bytes）；F5 只重载 Lua 不换原生 DLL。
